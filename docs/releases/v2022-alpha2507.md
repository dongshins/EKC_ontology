## 1. Executive summary (CHANGELOG를 보완하는 핵심 요약)
* Wiki 기반의 서술형 문서/규칙을 OWL 2 / RDF 1.1(Turtle) 산출물로 “재인코딩”하여,

  : (a) 기계 판독 가능성, (b) 추론 일관성, (c) 재현 가능한 배포/인용 기반을 확보.

* EKC를 Linked Open Vocabularies(LOV)에 등록하여,

  : 글로벌 시맨틱웹 참조 레지스트리에서 EKC 어휘의 발견가능성을 강화.

## 4. Protégé 운영 원칙: Domain/Range, 합집합(Union) 오해 대응
* Domain/Range의 “복수 선택”은 Protégé 내부에서 교집합으로 작동할 수 있음

  : 따라서 의도된 후보가 “합집합”일 때는 Domain/Range를 비워두고,

    `skos:scopeNote`로 의도한 후보를 명시하는 방식을 채택.

* 예시(운용 패턴):

  : “Domain : Record/Object/Event”

  : “Domain : Any/Index”

  : “Range : WebResource/Multimedia”

* 목적:

  : 추론/표현의 일관성을 확보하면서, 사용자에게는 의도된 적용 범위를 분명히 전달.

---

## 5. ‘Agent’ 처리: 불완전 정의(클래스 부재)와 의미 보존
* 현상:
  : EKC에서 일부 relation의 Domain/Range에 ‘Agent’가 언급되지만 class tree에는 없음.
  : 내용상 ‘기관’, ‘단체’ 등 Actor의 일부 하위범주를 가리키는 용례로 해석됨.
* 조치:
  1. Domain/Range는 상위 ‘Actor’로 포괄 설정
  2. 의미 보존을 위해 `skos:scopeNote`로 “Agent(Actor의 subclasses 중 기관, 단체 등)”을 주석으로 명시
* 예시:
  : `hasAffiliation`의 Range를 Actor로 설정하되, scopeNote에 Agent 의미를 남김.

---

## 6. Annotation 규칙 정비: HistoryNote, scopeNote, label
* skos:note의 확장 운용:
  : Notes뿐 아니라 변경 이력 기록용 “HistoryNote”에도 활용
  : 표기 규칙: skos:note 값 앞에 `[HistoryNote]` 글머리 부착
* rdfs:label의 운용 필요성:
  : URI/name은 유지하면서 “표시명(정책/용례/다국어)”을 유연하게 바꾸기 위해 label을 적극 사용
  : 예: 국가 정책에 따른 문화재 명칭 변화는 label 변경으로 대응
* 괄호 제거(명칭 위생):
  : instance 명칭에서 괄호가 오류를 유발하는 사례를 고려하여,
    class 명칭도 괄호를 제거하는 방향으로 정비
  : 단, rdfs:label에는 괄호를 포함해 엄밀성/가독성을 보완할 수 있음.

---

## 7. Relation naming / lexical enrichment (용어·의미 보완)
* relation 명칭 오류 수정(예: 수태지 의미의 표준화)
* in-law 계열 의미 확장:
  : “…BrotherInLaw…” / “…SisterInLaw…” / “…FatherInLaw…” 등에서
    실제 한국어 친족어휘(처남/시형제/형수/제수/올케/처형제/시부/시모 등)를 반영하도록
    의미 및 주석을 보강.
* `hasContextualElement` 기술문구 조율:
  : 위키 문서 간 상이한 설명을 조율하여 정의문을 통일.

---

## 8. Property axioms 실험과 “취소” 기록(중요)
* 일부 object property 간 “상호 subProperty” 실험을 수행했으나 최종 취소
  : ∵ Protégé UI/표현에서 상위 그룹(A1 등) 밖으로 빠져 가독성과 체계 인지가 악화
  : 결과적으로 온톨로지 사용/교육/유지보수에 혼란을 유발할 가능성이 큼.
* 향후 원칙(메모 기반 운영 가이드):
  [안A] 동치/역관계가 아닌 경우에는 별도 property로 유지하고,
       인지자의 맥락 구분에 의지하되 `skos:scopeNote`로 상단에 주석을 남긴다.
  [안B] 필요 시 N-ary relation + Enumerated class로 엄밀화(단, 배포물 포함 여부는 별도 결정).

---

## 9. Known limitations / non-goals
* Advanced/실험 패턴(N-ary, Enumerated 확장 등)은 “설계 메모/후속 버전 후보”로 분리 관리
  : 본 릴리즈 산출물에 실제 포함된 것과 “검토만 한 것”을 혼동하지 않도록,
    적용 여부를 항목별로 명시하는 것을 원칙으로 함.

---

## 10. Migration notes (사용자 관점)
* 일부 IRI/label 정비로 인해 SPARQL 패턴의 상수(IRI/label 의존부)가 영향을 받을 수 있음.
* Domain/Range를 “강제 제약(constraint)”으로 쓰지 않는 OWL의 성격(OWA)을 전제로,
  : 데이터 무결성 제약은 별도 SHACL/검증 파이프라인에서 수행하는 것을 권장.
