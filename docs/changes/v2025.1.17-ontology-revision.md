## 1. Diff at a glance

### Changed
- `owl:versionInfo` updated from `EKC 2025 [v2025.1.0]` to `EKC 2025 [v2025.1.17]`
- Ontology-header `dcterms:issued` changed from `xsd:dateTime` to `xsd:date`
- Ontology-header `dcterms:modified` updated to the v2025.1.17 patch date and changed from `xsd:dateTime` to `xsd:date`
- `dcterms:issued` range changed from `xsd:dateTime` to `rdfs:Literal`
- `dcterms:modified` range changed from `xsd:dateTime` to `rdfs:Literal`
- Added `rdfs:label` for `rdfs:seeAlso`
- Added `rdfs:label` for `dcat:contactPoint`
- Added explicit datatype declaration for `xsd:date`
- EKC term declarations are now serialized with explicit `ekc:` prefix notation

### Removed / simplified
- Removed local `owl:TransitiveProperty` typing from `owl:sameAs`
- Removed local `skos:scopeNote` on `owl:sameAs`
- Removed local redeclaration/documentation blocks for `skos:mappingRelation`, `skos:exactMatch`, and `skos:closeMatch`

### Not changed
- EKC class set (`owl:Class`): no change
- EKC namespace IRI: no change
- Version-line backbone (`owl:versionIRI`, `owl:priorVersion`, `dcterms:replaces`): retained
- Core EKC schema architecture: no class-level refactoring

## 3. Identity / mapping re-positioning

### 3.1 `owl:sameAs`

The local `owl:sameAs` declaration remains present, but it was simplified.

- BEFORE:
  - marked as `owl:SymmetricProperty` **and** `owl:TransitiveProperty`
  - Korean description emphasized strict identity only
  - included a restrictive `skos:scopeNote` directing users toward `skos:exactMatch` / `skos:closeMatch`
- AFTER:
  - marked as `owl:SymmetricProperty` only
  - Korean description shortened to `~ņÖĆ Ļ░Öļŗż`
  - example and note simplified
  - no local `skos:scopeNote`

### 3.2 `skos:mappingRelation`, `skos:exactMatch`, `skos:closeMatch`

In v2025.1.0, the TTL contained local EKC-side redeclarations/documentation blocks for these standard SKOS mapping properties.

In v2025.1.17, those local blocks were removed from the published EKC TTL.

### 3.3 Interpretation note

This release does **not** erase the SKOS vocabulary itself. Nor does it imply that identity/mapping policy became unimportant.

Rather, the EKC core TTL now embeds **less project-local explanatory policy** for those predicates. The practical implication is that stricter or more localized guidance may be maintained in derivative documentation or in a derivative ontology line, instead of being republished inside the EKC core TTL itself.

### 3.4 Relation to EKCD

Within the broader project context, this repositioning is consistent with maintaining an **EKCD (EKC-derived) ontology line** for derivative or project-local extensions.

Accordingly, v2025.1.17 can be read as reinforcing the distinction between:
- **EKC** = stable core ontology line and citable base release series
- **EKCD** = derivative extension space for stronger local semantic guidance and related project-level elaboration

The file was also cleaned up at the serialization level.

- EKC terms previously serialized with the local default prefix `:` are now serialized with explicit `ekc:`
- This does **not** change the EKC IRIs themselves; it is a notation/serialization cleanup only
- The leading BOM present in the earlier file is absent in the new file export

## 5. Compatibility / migration notes

- No class-set migration required
- No EKC IRI migration required
- Review any tooling that validates ontology-header dates as `xsd:dateTime` only
- Review any downstream workflow that expected local TTL-side declaration blocks for:
  - `skos:mappingRelation`
  - `skos:exactMatch`
  - `skos:closeMatch`
- Review any policy note, documentation, or training material that relied on the stricter local wording previously attached to `owl:sameAs`
- If exact reasoning behavior depended on the local `owl:TransitiveProperty` declaration for `owl:sameAs`, re-check that assumption explicitly
- If your project depends on derivative/local extension policy rather than only on the EKC core line, check whether that policy should now be maintained in EKCD-side materials instead of EKC-side TTL annotations

## 6. Release positioning

v2025.1.17 should be understood as a core-line publication cleanup patch rather than a schema redesign. @en

It is narrower than v2025.1.0 in scope, but it makes a meaningful difference in **where** local semantic policy is expressed: less inside the published EKC core TTL, and more appropriately in derivative/project-local extension layers where needed. @en
